A expressão que você quer simplificar é $F = \bar{x}z + y\bar{z}$.
A questão, na verdade, pede o contrário: qual soma de mintermos canônica se simplifica para a expressão $F = \bar{x}z + y\bar{z}$?
A expressão $F = \bar{x}z + y\bar{z}$ tem três variáveis: $x$, $y$, e $z$. Para encontrar a soma de mintermos, precisamos garantir que cada termo produto ($ \bar{x}z$ e $y\bar{z}$) contenha todas as variáveis.
                           1. Termo $ \bar{x}z$: Falta a variável $y$. $$\bar{x}z = \bar{x}z \cdot (y + \bar{y}) = \bar{x}zy + \bar{x}z\bar{y}$$ Organizando as variáveis na ordem $x, y, z$: $$\bar{x}yz + \bar{x}\bar{y}z$$
                           2. Termo $y\bar{z}$: Falta a variável $x$. $$y\bar{z} = y\bar{z} \cdot (x + \bar{x}) = xy\bar{z} + \bar{x}y\bar{z}$$

A questão pede o número médio de escritas em uma linha de cache antes de ser substituída (retirada) para que a política write-back seja mais eficiente que a write-through.
A eficiência é medida pelo tempo gasto em operações de escrita na memória principal. Para que o write-back seja mais eficiente, o tempo total que ele gasta para escrever a linha de volta na memória principal deve ser menor do que o tempo total que o write-through gasta para realizar todas as escritas individuais na memória principal.
________________
Análise dos Tempos de Escrita
1. Tempo de Escrita para Write-Through
No write-through, cada escrita na cache também é imediatamente escrita na memória principal.
                           * Tamanho da palavra: $8$ bytes.
                           * Tempo de transferência de $8$ bytes (uma palavra): $20\text{ ns}$.
Se houver $N$ escritas em uma linha antes de ser substituída: $$\text{Tempo}{\text{WT}} = N \times (\text{Tempo para escrever } 8\text{ bytes na memória principal})$$ $$\text{Tempo}{\text{WT}} = N \times 20\text{ ns}$$
2. Tempo de Escrita para Write-Back
No write-back, a escrita é feita apenas na cache. A escrita na memória principal (chamada write-back) só ocorre uma única vez quando a linha é substituída, se a linha foi modificada (assumindo que a linha foi escrita pelo menos uma vez, conforme o enunciado).
                           * Tamanho da linha de cache: $16$ bytes.
                           * A linha modificada de $16$ bytes precisa ser escrita de volta na memória principal.
                           * A linha de $16$ bytes corresponde a $\frac{16\text{ bytes}}{8\text{ bytes/palavra}} = 2$ palavras de $8$ bytes.
O tempo para transferir $16$ bytes (a linha inteira) é o tempo para transferir $2$ palavras de $8$ bytes: $$\text{Tempo de transferência de } 16\text{ bytes} = 2 \times (\text{Tempo para escrever } 8\text{ bytes na memória principal})$$ $$\text{Tempo de transferência de } 16\text{ bytes} = 2 \times 20\text{ ns} = 40\text{ ns}$$
Se a linha foi modificada (o que é garantido pelo enunciado), o tempo total gasto na memória principal para o write-back é: $$\text{Tempo}{\text{WB}} = \text{Tempo de transferência da linha de } 16\text{ bytes}$$ $$\text{Tempo}{\text{WB}} = 40\text{ ns}$$
________________
Comparação de Eficiência
Para que o write-back seja mais eficiente do que o write-through, o tempo total que ele gasta na memória principal deve ser menor: $$\text{Tempo}{\text{WB}} < \text{Tempo}{\text{WT}}$$
Substituindo as expressões: $$40\text{ ns} < N \times 20\text{ ns}$$
Isolando $N$: $$N > \frac{40\text{ ns}}{20\text{ ns/escrita}}$$ $$N > 2$$
O número médio de vezes ($N$) que a linha precisa ser escrita deve ser maior que 2 para que o write-back seja mais eficiente (ou seja, gaste menos tempo na memória principal) do que o write-through.
________________
Conclusão e Seleção de Opção
O menor número inteiro de escritas que torna o write-back mais eficiente é $N=3$. No entanto, as opções fornecidas são: A) 4, B) 8, C) 12, D) 16, E) 32. Todas as opções listadas são $\ge 4$, o que satisfaz a condição $N>2$.

Portanto, a opção correta é aquela que representa o cenário em que o write-back claramente é mais eficiente e, entre as opções, a menor que atende à condição de que $N$ deve ser maior que 2.

A linguagem $L = {w \in {a, b}^* | w \text{ não pode terminar com } ba}$ é a linguagem de todas as strings sobre o alfabeto ${a, b}$ que não terminam com o sufixo $ba$.
Vamos analisar as opções:
A) $\neg( (a \cup b)^*ba )$ Esta expressão regular, usando o símbolo de negação ($\neg$), denotaria o complemento da linguagem de todas as strings que terminam em $ba$. A negação de uma ER não é uma operação padrão em ERs e geralmente é usada para denotar o complemento da linguagem. Se fosse uma operação de complemento, denotaria a linguagem de todas as strings que não terminam em $ba$, que é exatamente $L$. No entanto, o complemento não é geralmente representado por $\neg$ em Expressões Regulares padrão (ERs). Além disso, a negação de uma ER não resulta necessariamente em outra ER; mas como as linguagens regulares são fechadas sob complemento, o complemento é sempre regular. Se a notação $\neg$ for aceita como o complemento, esta seria a representação da linguagem $L$.
B) $(a \cup b)^*ba$ Esta ER denota a linguagem de todas as strings que terminam com $ba$. Isso é o complemento de $L$ (exceto para o caso da string vazia $\lambda$ e strings de tamanho 1, que não terminam em $ba$).
C) $(a \cup b)^* - ( (a \cup b)^ba )$ Esta notação usa a diferença de linguagens, $L_1 - L_2$, onde $L_1 = (a \cup b)^$ (todas as strings) e $L_2 = (a \cup b)^ba$ (strings que terminam em $ba$). A linguagem resultante, $L_1 - L_2$, é a linguagem de todas as strings que estão em $(a \cup b)^$ mas não em $(a \cup b)^*ba$. Isso significa todas as strings que não terminam em $ba$, que é exatamente a linguagem $L$. Embora a diferença de linguagens $L_1 - L_2$ seja uma operação válida no contexto de linguagens e linguagens regulares (pois linguagens regulares são fechadas sob diferença), não é estritamente uma Expressão Regular na notação padrão (que só usa $\cup, \cdot, *$).
D) $( (a \cup b)^* (b \cup aa) ) \cup a \cup \lambda$ Vamos tentar descrever o conjunto $L$ de uma maneira construtiva com uma ER padrão. A linguagem $L$ consiste em todas as strings que não terminam com $ba$. Isso significa que o final da string pode ser:
                           1. $\lambda$ (string vazia)
                           2. $a$
                           3. $b$
                           4. $aa$
                           5. $ab$
                           6. Qualquer string terminada em $a$ (exceto $\lambda$ e $a$)
                           7. Qualquer string terminada em $b$
A única terminação proibida é $ba$. Se uma string $w$ não termina em $ba$, ela deve ser uma das seguintes formas:
                           * A string vazia: $\lambda$
                           * A string termina em $a$: $(a \cup b)^*a$
                           * A string termina em $b$: $(a \cup b)^*b$
                           * A string é de comprimento 1: $a$ ou $b$.
                           * A string é de comprimento 0: $\lambda$.
Se a string $w$ tem comprimento maior ou igual a 2, ela não termina em $ba$ se:
                           * Termina em $bb$: $(a \cup b)^*bb$
                           * Termina em $aa$: $(a \cup b)^*aa$
                           * Termina em $ab$: $(a \cup b)^*ab$
Qualquer string $w \in L$ deve pertencer a:
                           * Caso 1: $w$ não tem o sufixo $b$. Isso significa que $w$ é $(a \cup b)^*$, mas o caractere anterior ao último não é $b$. Isso é difícil de expressar.
                           * Caso 2: A string $w$ termina com $b$. Se termina em $b$, não pode terminar em $ba$. $w = (a \cup b)^*b$.
                           * Caso 3: A string $w$ termina com $a$. Se termina em $a$, não pode terminar em $ba$. $w = (a \cup b)^*a$.
                           * Caso 4: A string $w$ é $\lambda$.
A união de $(a \cup b)^*b$ e $(a \cup b)^*a$ cobre todas as strings de comprimento $\geq 1$. $(a \cup b)^*a \cup (a \cup b)^*b = (a \cup b)^+(a \cup b)$. Essa união é simplesmente $(a \cup b)^+$.
No entanto, a terminação em $a$ ou $b$ não garante que a string não termina em $ba$.
                           * Exemplo: $w=aba$. $aba \in (a \cup b)^*a$. $aba$ termina em $ba$ não $a$. $aba \notin L$.
Uma string $w$ não termina em $ba$ se:
                           1. $w$ tem comprimento $< 2$. $w \in {\lambda, a, b}$.
                           2. $w$ tem comprimento $\geq 2$, e: a. O penúltimo é $a$. Então, $w = (a \cup b)^*a(a \cup b)$.

                           * Se o último é $b$: $w'ab$. OK. b. O último é $b$. Então, $w = (a \cup b)^*b$. (Já coberto pelos casos de $a$ e $b$ como último caractere).
A união: $L = (a \cup b)^*a \cup (a \cup b)^*b \cup {\lambda}$ não funciona, pois, por exemplo, $aba$ termina em $a$ e $aba \notin L$.
Uma descrição mais precisa: $L$ é a união de:
                           1. Strings de comprimento 0 ou 1: $\lambda \cup a \cup b$.
                           2. Strings de comprimento $\geq 2$ que não terminam em $ba$.

                           * Strings que terminam em $ab$: $(a \cup b)^*ab$. (Isso inclui $ab, aab, bab, aaab, baab, ...$)

1. Assertiva: A gerência efetiva de projetos de software deve abranger três aspectos, a saber, produto, processo e projeto.
                           * Conceito-Chave: O gerenciamento de um projeto de software é uma tarefa complexa que precisa conciliar as restrições e objetivos relacionados a três elementos cruciais:
                           * Produto: O software que será entregue (o que precisa ser feito).
                           * Processo: O conjunto de atividades, tarefas e práticas para construir o produto (como será feito).
                           * Projeto: O planejamento, controle e organização do trabalho (quem, quando e quanto custa).
                           * Análise: Uma gerência eficaz deve, de fato, equilibrar esses três elementos para garantir a entrega bem-sucedida do software.
                           * Resultado: (V) Verdadeira
2. Assertiva: A análise de risco de um projeto visa a analisar a viabilidade de produzir software dentro do custo e esforço estimados.
                           * Conceito-Chave:
                           * Análise de Risco: Identificar, analisar e planejar a mitigação de eventos potenciais que podem impactar negativamente o projeto (custo, cronograma, escopo, qualidade, etc.).
                           * Análise de Viabilidade/Escopo: Determinar se o projeto é possível e justificável em termos técnicos, financeiros e operacionais antes de iniciar a estimativa detalhada e o planejamento de risco.
                           * Análise: Embora o risco de exceder o custo e o esforço seja um dos riscos a serem gerenciados, o propósito principal da análise de viabilidade é o de determinar se o projeto deve ser iniciado ou continuado, avaliando se ele pode ser concluído com sucesso e se vale a pena ser feito, o que é um passo anterior e distinto da análise de risco. A análise de risco lida com a incerteza, não com a viabilidade básica.
                           * Resultado: (F) Falsa
3. Assertiva: O desenvolvimento de cronogramas de projetos pode ser desenvolvido com o auxílio de modelos como PERT (Program evaluation and review technique) ou CPM (Critical Path Method).
                           * Conceito-Chave: PERT (Técnica de Avaliação e Revisão de Programas) e CPM (Método do Caminho Crítico) são técnicas clássicas e amplamente utilizadas para planejamento, programação e controle de projetos. Elas ajudam a determinar a duração total do projeto, a identificar as atividades críticas e a gerenciar a alocação de tempo.
                           * Análise: Ambas são ferramentas essenciais para o desenvolvimento de cronogramas e gráficos de Gantt, sendo aplicáveis a projetos de software.
                           * Resultado: (V) Verdadeira
